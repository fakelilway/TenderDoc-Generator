# 团队协作规范

本文档用于规范 TenderDoc-Generator 团队的协作流程，目标是降低分支冲突、保证代码质量，并让每个 Minitask 的推进过程可追踪、可审查。

---

## 1. 基本原则

- 每个 Minitask 开始前，必须先创建一个 Issue。
- 每个 Issue 对应一个独立分支，避免多人在同一分支上直接开发。
- 每次提交的改动尽量保持单一目标，避免一个 PR 混入多个无关任务。
- 任何代码合并都必须经过至少 1 名其他成员 review。
- 未 review 的 PR 不得直接 merge 到主分支。

---

## 2. 标准流程

### 2.1 创建 Issue

在开始一个 Minitask 之前，先创建 Issue，Issue 中至少包含以下内容：

- 任务名称
- 对应的 Minitask 编号
- 目标与验收标准
- 预期修改文件范围
- 风险点和依赖项

建议使用统一标题格式：

- `M12: 实现解析 Agent`
- `M18: 实现检索器`
- `M24: 集成生成与导出`

### 2.2 创建分支

每个 Issue 由独立分支承载，建议命名格式：

- `feature/m12-parser-agent`
- `feature/m18-retriever`
- `feature/m24-export-pipeline`

分支命名要求：

- 使用小写字母
- 使用短横线分隔单词
- 分支名中包含对应 Minitask 编号
- 禁止多人在同一 feature 分支上并行开发

### 2.3 开发与提交

开发时遵循以下规则：

- 先拉取最新主分支，再从主分支创建新分支。
- 开发过程中尽量只修改本任务范围内的文件。
- 提交信息要清晰，例如：
  - `feat: add parser agent schema`
  - `fix: handle docx extraction edge case`
  - `test: add parser accuracy test`
- 单次提交尽量小而清晰，便于 review 和回滚。

### 2.4 提交 Pull Request

代码完成后，必须创建 Pull Request，PR 中应包含：

- 关联的 Issue 编号
- 改动摘要
- 测试结果
- 影响范围
- 是否需要数据库或配置变更

PR 标题建议格式：

- `[M12] 实现解析 Agent`
- `[M18] 检索器与向量检索接入`

### 2.5 Review 与合并

- 至少 1 名未参与该分支开发的人进行 review。
- Review 重点检查：逻辑正确性、测试覆盖、命名一致性、是否引入冲突风险。
- review 通过后，才能 merge。
- 合并方式优先使用 `squash merge`，减少主分支提交噪音。

---

## 3. 防止分支冲突的规则

为了尽量避免 branch 冲突，团队必须遵守以下规范：

### 3.1 先同步再开发

每次开始新任务前，先执行：

```bash
git checkout main
git pull --rebase
```

然后再从最新主分支创建功能分支。

### 3.2 单人单分支

- 一个分支只允许一个人负责。
- 不要在同一个分支上多人同时改同一类文件。
- 如果两个任务会改同一文件，必须先协调顺序，避免互相覆盖。

### 3.3 尽量减少大范围改动

- 一次只改一个功能点。
- 避免格式化整个仓库。
- 避免在一个 PR 中同时做重构、功能、文档大改。

### 3.4 高频同步主分支

如果主分支持续更新，开发分支应定期同步：

```bash
git fetch origin
git rebase origin/main
```

遇到冲突时优先本地解决，解决后再继续 rebase。

### 3.5 修改共享文件前先协调

以下文件属于高冲突风险文件，修改前应先确认是否有其他人正在编辑：

- `README.md`
- `setup.md`
- `docker-compose.yml`
- `minitasks.md`
- 公共配置文件

如果必须修改共享文件，建议拆成更小的 PR。

---

## 4. 推荐工作流

建议按下面的节奏执行：

1. 创建 Issue
2. 从最新 `main` 拉出 feature 分支
3. 开发并提交小粒度 commits
4. 本地自测通过
5. 提交 PR
6. 由另一位成员 review
7. Review 通过后 merge
8. 删除 feature 分支

---

## 5. PR 检查清单

每个 PR 提交前，请确认：

- [ ] 已关联 Issue
- [ ] 分支来自最新 `main`
- [ ] 改动范围单一清晰
- [ ] 已通过本地测试
- [ ] 已更新必要文档
- [ ] 已经有至少 1 人 review
- [ ] 不包含无关文件修改
- [ ] 不会与其他进行中的分支冲突

---

## 6. 建议的团队约定

- `main` 分支始终保持可用。
- 不允许直接向 `main` 提交代码。
- 如果任务较大，先拆分 Issue，再拆分 PR。
- 任何涉及数据库表结构、环境变量、docker 配置的改动，都必须在 PR 描述里明确说明。
- 如发现两个分支会修改同一文件的同一部分，优先合并一个，再开始另一个。

---

## 7. 最小冲突实践

为了减少冲突，建议：

- 先改新文件，再改公共文件。
- 配置文件尽量集中管理。
- 文档改动与代码改动尽量分开提交。
- 频繁 rebase，而不是长期落后于主分支。

---

## 8. 责任边界

- 开发者对自己的分支负责。
- Reviewer 对 PR 的质量负责。
- 合并者对主分支稳定性负责。
- 如果发现冲突风险，必须先暂停合并并协调。

---

## 9. 默认模板

### Issue 模板

- 标题：`[M12] 实现解析 Agent`
- 内容：目标、验收标准、影响文件、测试方式、风险

### PR 模板

- 标题：`[M12] 实现解析 Agent`
- 内容：关联 Issue、改动摘要、测试结果、截图或日志、注意事项

---

## 10. 执行要求

这份规范对所有团队成员都适用。任何人开始一个 Minitask 前，必须先开 Issue；完成后必须提 PR；PR 必须经过 review 后才能 merge。对于容易冲突的公共文件，尽量通过协调和拆分任务来降低风险。
