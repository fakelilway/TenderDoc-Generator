# 标书生成边界约定

本项目只保留一个章节结构来源，避免模板、Prompt、脚本和知识库互相抢控制权。
本文件同时是产品范围的权威说明（原 `docs/zhengqi_product_scope.md` 已合并到这里）。

## 产品范围

当前第一版面向正奇建设的市政/公路投标业务，不按全行业通用标书软件设计。生成器、风格案例推荐、知识库标签和质量评估优先服务：

- 市政工程：市政道路、排水、管网、道路附属设施、市政维修/养护/改造。
- 公路工程：公路改建、扩建、维修养护，路基、路面、桥涵附属及道路安全相关内容。
- 交通安全设施：标志标线、护栏、防眩、隔离栅、轮廓标、道口标柱及其养护、维修、更换。

第一版不主动覆盖：房建总承包、园林绿化、水利水电、机电安装、政府采购服务、纯货物采购、勘察设计/监理/咨询。这些方向保留扩展可能，但不能成为默认风格案例、默认 prompt 或默认评估目标。

知识库优先录入：正奇已中标投标文件、公司资质/人员证书/业绩、市政道路与公路改扩建施工方案、交安专项方案、常见废标条款和评分办法响应口径。

第一版验收不以“支持多少行业”为标准，而以正奇真实项目可用性为标准：能识别正奇主线项目类型、推荐对应风格案例、独立生成三卷、DOCX 接近历史中标文件格式、审查覆盖废标/资质/评分项、用户可编辑确认下载。

## 结构来源

`TenderRequirements` 表示招标文件要求：资格、评分点、废标条款、工期、保证金和投标文件格式等。它回答“必须响应什么”，其中 `bid_format_requirements` 是卷册、表单、签字盖章、正副本、密封/电子标等格式要求的人工确认入口。

线上结构权威是：**招标文件格式要求 + 人工确认目录**。招标文件中明确规定的投标文件组成、格式要求和表单清单（通常在“投标文件格式”“投标文件的编制”“密封和标记”等章节）是最高权威，生成前必须由制作员确认；招标全文的格式相关段落会进入长上下文 prompt。

`BidTemplate`/`TemplateProfile` 在线上不再是默认结构权威。历史投标 PDF 会保留为**公司风格案例**：用于参考施工组织设计深度、表格习惯、图片位、禁用语气和正奇表达方式。只有用户主动选择案例时，它才作为风格参考进入生成；没有选择案例时，系统不会自动加载默认模板。

系统同时维护两种目录：

- `bid_outline_json`：技术标/施工组织设计写作大纲，用于 RAG 检索和技术章节生成。
- `document_outline_json`：完整标书目录，用于展示和约束整个投标文件包，包含商务/资格、技术、报价/经济标和附表。
- `draft_volumes`：商务文件、技术文件、报价文件三卷 Markdown，是分卷预览和单卷下载的优先来源。

仓库中的公路样本只保留给离线评估、脚本调试和风格案例导入，不再自动套到新项目。新项目默认按招标文件格式要求和人工确认目录生成。

生成链路应先产出商务、技术、报价三个独立分卷，再按用户选择合并为完整投标文件。不得只生成一份完整稿后在下载阶段强行截断为分卷。

## 写作来源

Generator prompt 只约束角色、文风、真实性边界和禁止事项。它不能再内置另一套目录或表单模板。

知识库/RAG 只提供历史材料、技术措施、企业表达习惯和可参考措辞。RAG 片段不能改变章节结构，不能替代招标文件格式要求，也不能把样本里的真实人员、电话、身份证、金额等当作当前项目事实。

## 输出排版

Markdown 负责内容层级和正文。DOCX 的视觉排版、封面、目录、页眉页脚、字体、间距和分册策略统一由 `backend/utils/docx_exporter.py` 负责。

## 脚本定位

`scripts/` 和 `backend/scripts/` 下的命令行工具是离线分析、评估、调试和初始化入口，不属于线上生成链路。线上链路应从 API/service 层进入：解析招标需求，确认投标文件格式和目录，可选选择风格案例，检索知识材料，生成 Markdown，再由 exporter 生成 DOCX。

## 历史案例资产

仓库内置历史案例样本位于 `backend/templates/bid_templates/road_first_envelope_template.json`。本地或部署环境可用下面命令幂等导入公司风格案例库：

```bash
python backend/scripts/seed_default_template.py
```

导入后，新项目仍需要人工主动选择该案例才会参考其风格；未选择案例时不会自动加载默认模板。
