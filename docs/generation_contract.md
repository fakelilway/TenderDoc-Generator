# 标书生成边界约定

本文件定义 TenderDoc-Generator 当前线上链路的职责边界。目标是避免模板、Prompt、脚本、知识库和导出器同时抢“格式控制权”。

## 一句话原则

招标文件中的“投标文件格式/响应文件格式”是最高权威。系统只复制原格式、填有依据的字段、写技术正文、审查风险；不得生成一套自认为合理的商务或报价格式。

## 结构来源

当前结构来源只有三类：

1. 招标文件原始格式页：商务文件、技术文件、报价文件的目录、函件、表格、签章位、下划线和附件说明。
2. `format_outline_tree`：Parser Agent 提取的三卷目录树，用于导航、技术标题收集和人工确认。
3. 人工确认结果：用户在前端确认或修正后的解析 JSON、大纲和资料选择。

不再作为结构来源：

- 默认模板。
- 公司风格案例。
- 知识库 RAG 片段。
- Generator Prompt 内置目录。
- 离线脚本分析结果。

## 生成职责

| 模块 | 可以做什么 | 不能做什么 |
|------|------------|------------|
| Parser Agent | 提取项目事实、资质、评分项、废标项和 `format_outline_tree` | 决定最终 DOCX 版式 |
| Original Format Service | 从 DOCX/PDF 招标文件复制格式页 | 重画表格、改签章位、删下划线 |
| Form Filler | 替换有证据的字段，例如项目名、招标人、投标人、工期、质量 | 编造证号、金额、人员、日期 |
| Content Writer | 在技术正文区域写施工组织方案 | 改商务/报价函件和表格结构 |
| V2 Audit | 挡格式、内容和证据问题 | 为了通过而自动造假 |
| Reviewer Agent | 审废标、资质、报价人工确认和响应性风险 | 代替人工最终盖章确认 |
| DOCX Exporter | 普通 Markdown 排版、图片和表格渲染 | 覆盖招标文件原格式页 |

## 格式处理规则

- DOCX 招标文件：使用 `python-docx` 读取源文件，将格式章 OOXML 元素复制到目标 DOCX。
- PDF 招标文件：使用 PyMuPDF 渲染格式页整页图片，按 PDF 页面尺寸放入 DOCX，并叠加可编辑文本层。
- PDF 页会写入不可见页标记，导出拆卷时按整页移动，避免页图和文本层被拆散。
- 商务文件和报价文件原则上只做有限字段替换，未知信息保留下划线。
- 技术正文可以追加或填入技术卷，但不能破坏招标文件给出的技术卷标题和顺序。
- 格式复制、技术正文写作或审查失败时，系统必须报错并保留失败原因。

## 审查处理规则

V2 内置审查位于 `backend/services/v2_audit_service.py`：

- 格式层：检查表格是否被拍扁、签章位是否丢失、下划线是否保留、图表/图片要求是否落实。
- 内容层：检查施工方案是否过短、是否出现 AI 元话语、金额、身份证号等高风险内容。
- 证据层：检查填入字段是否与公司档案一致，关键字段是否缺失。

工作流审查位于 `backend/agents/reviewer_agent.py`：

- 规则审查覆盖废标条款、资质要求、投标保证金、报价人工确认等。
- 可选 LLM 审查只补充人工可读的风险点，不能覆盖规则审查结论。

## 知识库边界

知识库负责提供证据和素材：

- 公司证件、人员证件、业绩、社保、资质、安全生产许可证等作为事实依据。
- 施工方案、组织机构图、进度计划图、施工总平面图等作为技术正文和图片候选。
- 历史中标标书作为风格与质量基线，不作为当前项目格式。

知识库不得改变招标文件目录，也不得把其他项目的人员、证号、金额、电话、日期自动填到当前项目。

## 公司风格案例边界

公司风格案例只影响：

- 技术正文深度。
- 正奇常用表达。
- 禁用语气。
- 图表/图片位参考。
- 真实中标标书质量对比。

公司风格案例不影响：

- 商务文件目录。
- 报价文件目录。
- 投标函格式。
- 资格审查表格。
- 签字盖章位置。

## 交付边界

TenderDoc-Generator 输出：

- 完整 DOCX。
- 商务/技术/报价三卷 DOCX。
- Markdown。
- 审查报告。
- 资料引用和人工确认点。

新点投标文件制作软件负责：

- 最终电子投标文件制作。
- CA 签章。
- 加密。
- 上传。
- 平台校验。
