## 已对齐的原则

1. 不要 fallback——生成失败就报错，不造假
2. 格式是最高权威——招标文件格式章节 = 标书的框架
3. 不要为了过而过——查根因再改
4. 删代码前 grep 引用
5. LLM 不应该管结构——结构交给代码

## Codex 的下一步建议

### M1 Skeleton MVP — 已本地完成，待端到端验证

**目标:** 把 Generator 从"LLM 生成整卷(结构+内容)"改成"代码铺骨架 → LLM 只填槽位"

**已遵守的不改范围:**
- Parser / Knowledge Base / Workflow Service / Frontend / DOCX Export — 全不变
- `generate_bid_package()` 入口不变

**已加的:**
- `render_all_volume_skeletons()` / `render_volume_skeleton()` / `expected_volume_titles()`
- Content Fill Agent 的 prompt 从"按这些节点生成整卷"改成"在这个已有骨架的节点下填内容"

**期望效果:**
- LLM 不再自己决定标题结构；缺节点会被确定性预审打回。
- 结构审计仍保留 LLM Pass 1，用于层级、归属卷、多余节点等更复杂判断。

### M2 原文模板抽取 — 已本地完成，待端到端验证

**已加的:**
- `extract_format_template_blocks()`：按三卷格式树顺序，从 tender_text 的格式章节抽取叶子节点原文块。
- 骨架渲染优先嵌入招标文件原文模板；抽不到才使用通用空白占位。
- Prompt 明确要求优先保留骨架里的原文模板，不改函件正文和表头。

**注意:** M2 目前处理的是 Markdown 骨架里的原文模板，不是 DOCX 字体字号/页边距/表格线的最终版式复制。版式级复刻要放到后续 DOCX exporter/模板映射阶段做。

### 下一步建议 (M3)

- 用用户前端“重新生成”跑一个真实项目，重点看三件事：
  1. 商务/技术/报价目录是否严格来自招标文件格式树。
  2. 是否还会出现投标函重复、漏节点、报价卷层级错位。
  3. 函件和表格是否优先来自招标文件格式章节原文，而不是通用表格。
  4. 失败时实时状态是否能明确指出缺哪个分卷、哪个节点。
- 如果 M2 结构和原文模板稳定，再做 M3：节点级填槽，而不是让 LLM 输出整卷 Markdown；这样可以更彻底防止它改标题/改表头。

## 用户发现的真实 Case

### Case 1: 长丰县罗塘乡（美丽宜居村庄建设）
- **文件**: 招标文件正文.pdf (892页)
- **类型**: 公路工程，常规施工标
- **卷册结构**: 商务文件(10表单) + 技术文件(施工组织设计+其他内容) + 报价文件(投标函+工程量清单)
- **格式树特征**: 商务8个根节点（资格审查资料有7个子项），技术1个根节点，报价2个根节点
- **特殊陷阱**: 无。标准三卷结构。
- **Parser质量**: 格式树提取准确
- **生成质量**: 多次结构审计失败（LLM多加子章节、漏节点），已通过M1骨架修复

### Case 2: 萧县2025农村公路（联网路工程）
- **文件**: 1招标文件正文(1).pdf (150页)
- **类型**: 公路工程，双信封制，技术评分最低标价法
- **卷册结构**: 第一信封(商务及技术文件，9大类表单+8个施工附表) + 第二信封(报价文件:投标函+已标价清单+其他资料)
- **格式树特征**: 商务8节点(含9子项资格审查)，技术1节点(施工组织设计，9个要点+8个附表)，报价3节点
- **特殊陷阱**: ⚠️ 投标函在两个信封各出现一次（合法，不能判越卷）；施工组织设计有硬性9要点+8附表；第一信封(商务+技术合并)不是传统三卷
- **Parser质量**: 格式树准确区分了双信封；source字段曾输出flat string→已归一化
- **生成质量**: 待测（刚加M1骨架）

### Case 3: 南陵县三里镇（联网路工程）
- **文件**: 招标文件正文(3).pdf (142页)
- **类型**: 公路工程，公开招标，合理低价法
- **卷册结构**: 第九章投标文件格式，28个表单，75个表格
- **格式树特征**: 与萧县结构相似，但评标办法不同（合理低价法 vs 技术评分最低标价法）
- **特殊陷阱**: ⚠️ 落实中小企业政策要求分包≥40%；"投标函"出现32次（含目录引用，需过滤）
- **Parser质量**: 待测
- **生成质量**: 待测

### Case 4: 颍州区袁集镇（道路排水改造）
- **文件**: 颍州区袁集镇福满路、孝梯路道路排水改造工程(定稿)(1).docx (69页)
- **类型**: 市政工程，竞争性磋商（非招标！）
- **卷册结构**: 第六章"响应文件格式"（非"投标文件格式"）
- **格式树特征**: DOCX格式，术语不同：供应商(非投标人)、响应文件(非投标文件)、磋商(非开标)
- **特殊陷阱**: ⚠️ 竞争性磋商格式，采购术语完全不同；Parser当前的"第X章 投标文件格式"正则无法匹配"响应文件格式"；DOCX文件需python-docx解析
- **Parser质量**: ⚠️ 当前Parser会漏掉"响应文件格式"章节
- **生成质量**: 未测

### Case 5: 鸠江区农村公路（日常养护）
- **文件**: 招标文件正文 (1)(1).pdf (71页)
- **类型**: 公路养护，竞争性磋商（非招标！）
- **卷册结构**: 第十章"响应文件格式"，预算344万，365日历天
- **格式树特征**: 养护类项目，非新建工程；评分体系不同（综合评分法）
- **特殊陷阱**: ⚠️ 同Case4——竞争性磋商，术语体系不同；公路养护资质要求（三类乙级/路基路面养护乙级），非施工总承包资质
- **Parser质量**: ⚠️ 同Case4
- **生成质量**: 未测
